маршрутизации спецтехники (ИС УТО)


Оптимизация маршрутизации спецтехники на нефтяных месторождения
1. Контекст и описание проблемы	
1.1. Бизнес-контекст
Компания эксплуатирует парк спецтехники на нескольких нефтяных месторождениях. Ежедневно формируются десятки заявок на выполнение работ на скважинах и других объектах. Диспетчер планирует заявки на неделю и более вперёд, назначает технику и водителей вручную, контролирует перемещения в реальном времени через систему мониторинга Wialon.
Дорожная сеть месторождений включает как официальные дороги, так и накатанные полевые пути, восстановленные из GPS-треков техники. 
1.2. Проблема
Ручное планирование приводит к системным потерям по нескольким направлениям:
• Диспетчер не учитывает фактическую дорожную сеть месторождения при назначении техники, ориентируясь на приблизительные расстояния.
• Не учитывается будущая занятость техники и расчётное время освобождения, что приводит к простоям и задержкам.
• Отсутствует механизм объединения заявок: возможность выполнить несколько близких заявок одной машиной за один выезд не используется.
• Приоритеты заявок обрабатываются неформально, без количественного учёта влияния на общую эффективность парка.
Последствия: избыточный пробег и перерасход топлива, простои техники, срыв сроков выполнения высокоприоритетных заявок, неэффективное использование парка в целом.
2. Цель проекта
Разработать прототип backend-сервиса, который решает задачу оптимальной маршрутизации спецтехники. Сервис должен:
1. Рекомендовать оптимальную технику под заявку с обоснованием выбора: расстояние по графу дорог, расчётное время прибытия (ETA), приоритет заявки, доступность техники.
2. Строить маршруты по графу дорог месторождения (граф уже восстановлен из GPS-треков и предоставляется в готовом виде).
3. Учитывать приоритеты заявок: для высокого приоритета допускается выбор более далёкой, но свободной техники; для низкого допускается ожидание ближайшей.
4. Оценивать многозадачность: анализ целесообразности объединения нескольких заявок одной машиной за один выезд.
5. Отдавать результат через REST API и поддерживать визуализацию маршрутов (минимально - координатами/геометрией, на демо - на карте).
3. Формализация задачи оптимизации
3.1. Тип задачи
Задача представляет собой вариант Vehicle Routing Problem (VRP) со следующими особенностями:
• Multi-Depot: каждая единица техники стартует из своего текущего местоположения (множество виртуальных депо, а не одна центральная база).
• Time Windows (VRPTW): каждая заявка имеет временное окно выполнения, определяемое датой и сменой (дневная/ночная).
• Multi-stop: одна машина может выполнить несколько заявок за один выезд, если это оптимально по суммарному пробегу и укладывается в SLA.
• Open-end routes: техника не возвращается на стартовую точку - она остаётся на месте последней выполненной заявки до следующего назначения.
• Fleet compatibility: не любая техника может выполнить любую заявку. Есть возможность создать словарь совместимости типов работ и типов техники.
3.2. Целевая функция
Минимизировать суммарную стоимость обслуживания всех заявок, которая складывается из:
• общего пробега всех машин (суммарная длина маршрутов по графу дорог);
• штрафов за нарушение временных окон и приоритетов (soft constraints, высокоприоритетные заявки штрафуются сильнее за опоздание);
• суммарного времени простоя техники.
3.3. Ограничения
6. Каждая заявка должна быть выполнена ровно одной единицей техники.
7. Временные окна: исполнение заявки может превышать запланированное время на ее выполнение.
8. Совместимость: техника назначается на заявку при совместимом типе работы.
9. Маршрут строится строго по графу дорог (не по прямой).
10. Техника остаётся на месте последней заявки (не возвращается на базу).
3.4. Свобода выбора подхода
Участники свободны в выборе методов решения. Возможные подходы включают, но не ограничиваются:
• VRP-солверы
• Классические эвристики.
• Метаэвристики.
• AI/ML.
• Гибридные подходы.
Приветствуется сравнение нескольких подходов и обоснование выбора.
4. Предоставляемые данные
Участники получают подключение к БД со следующими данными. Все координаты обезличены, но метрически согласованы между собой (расстояния между точками корректны).
4.1. Граф дорог: узлы (references.road_nodes)
Узлы дорожной сети месторождения, включая полевые дороги, восстановленные из GPS-треков. Каждый узел имеет уникальный идентификатор и координаты.
ПолеТипОписаниеidserial (PK)Технический ключnode_idinteger (unique)Логический идентификатор узла для построения графаlonnumeric(12,8)Долгота (обезличена, метрически согласована)latnumeric(12,8)ШиротаНазначение: вершины графа дорог; привязка позиций техники и скважин к ближайшему узлу (map-matching).
4.2. Граф дорог: рёбра (references.road_edges)
Рёбра (дороги) между узлами графа. Каждое ребро имеет начальный узел, конечный узел и вес (стоимость проезда).
ПолеТипОписаниеidserial (PK)Технический ключsourceintegerID начального узла (→ road_nodes.node_id)targetintegerID конечного узла (→ road_nodes.node_id)weightnumeric(12,6)Вес ребраСвязь: road_edges.source и road_edges.target ссылаются на road_nodes.node_id (логическая связь).
Назначение: построение графа для алгоритмов кратчайшего пути; расстояние маршрута = сумма weight рёбер пути.
4.3. Скважины / точки назначения (references.wells)
Справочник скважин и объектов работ. Координаты согласованы с графом дорог. Скважина является целевой точкой заявки: заявка содержит destination_uwi, по которому определяется местоположение.
ПолеТипОписаниеidserial (PK)Технический ключuwivarchar(50) (unique)Уникальный идентификатор скважиныlatitudenumeric(12,8)Широта (обезличена)longitudenumeric(12,8)Долгота (обезличена)well_namevarchar(255)Отображаемое название скважиныОграничение: (latitude, longitude) - либо обе заполнены, либо обе NULL.
4.4. Снапшоты техники (references.wialon_units_snapshot_1/2/3)
Три снапшота (среза) парка техники из системы мониторинга Wialon, сделанные в разные моменты времени. Одна и та же машина (wialon_id) встречается в разных снапшотах с разными координатами и временными метками, что позволяет проследить перемещения.
ПолеТипОписаниеwialon_idbigintИдентификатор единицы техники в Wialon (ключ)nmtextНазвание техникиcls, muintegerСлужебные поля Wialon (можно игнорировать)pos_tbigintTimestamp позиции (Unix epoch, секунды)pos_ydoubleШирота (Y-координата)pos_xdoubleДолгота (X-координата)registration_platetextГосномер (обезличен)payload_jsonjsonbПолный JSON снапшота Wialon (доп. данные)Назначение:
• Текущие позиции техники (стартовые точки маршрутов): pos_x, pos_y из последнего снапшота.
• Вычисление средней скорости техники: Δ(расстояние) / Δ(pos_t) между снапшотами одного wialon_id.
• Исторический анализ (при необходимости): кластеры стоянок, паттерны движения, типовые маршруты.

4.6. Словарь совместимости 
Словарь определяет, какая техника (или тип техники) может выполнять какой тип работ. 
Использование: при назначении техники на заявку необходимо проверить, что тип техники допустим для task_type данной заявки. Это может быть реализовано как жёсткое ограничение (запрет назначения) или как мягкое (штраф за несовместимость).

6. Взаимосвязи данных и flow построения маршрута
6.1. Цепочка привязки данных
Основной flow от исходных данных к маршруту:
Заявка → скважина: task.destination_uwi → references.wells.uwi → (longitude, latitude).
Скважина → узел графа: wells(lon, lat) → nearest road_nodes.node_id = end_node.
Техника → узел графа: wialon_snapshot*(pos_x, pos_y) → nearest road_nodes.node_id = start_node.
Маршрут по графу: road_edges(source, target, weight) → кратчайший путь start_node → end_node.
Результат: длина маршрута = Σ weight рёбер; время = длина / avg_speed.


7. Что нужно реализовать
Ниже описаны программные модули, которые участникам необходимо спроектировать и реализовать для работы системы.
7.1. Модуль загрузки и индексации графа
Модуль при старте сервиса загружает таблицы road_nodes и road_edges из БД и строит in-memory граф. На этом же этапе формируется пространственный индекс (KD-Tree или аналог) по координатам узлов, чтобы любая произвольная точка (позиция техники, координаты скважины) могла быть привязана к ближайшему узлу графа (map-matching). Результат: объект графа, готовый для запуска алгоритмов кратчайшего пути, и функция snap_to_node(lon, lat) → node_id.
7.2. Сервис кратчайших путей и матрица стоимостей
На основе загруженного графа необходимо реализовать сервис, который по паре узлов (start_node, end_node) возвращает: длину кратчайшего пути (сумма weight), расчётное время в пути (длина / средняя скорость), упорядоченный список узлов маршрута и их координаты. Для пакетных запросов (например, при расчёте назначений на смену) модуль должен уметь формировать матрицу расстояний и времён для множества пар точек - стартовые позиции всех рассматриваемых машин × точки всех заявок. Выбор графового алгоритма остаётся за участниками; оценивается корректность результата и скорость работы на предоставленном графе.
7.3. Модуль состояния парка техники
Модуль отвечает за актуальную картину доступности парка. На входе - снапшоты из Wialon и текущий план заявок; на выходе - для каждой единицы техники: текущий статус (свободна / занята), прогнозируемое время и место освобождения (узел графа, где машина окажется после завершения текущей заявки), список совместимых типов работ (skills). Этот модуль не принимает решений о назначении - он поставляет данные для модулей скоринга и оптимизации. При изменении плана (новое назначение, отмена заявки) состояние парка должно пересчитываться.
7.4. Ядро оптимизации (решатель)
Центральный модуль системы, реализующий логику назначения техники на заявки. Принимает на вход матрицу стоимостей (7.2), состояние парка (7.3) и набор заявок с параметрами из раздела 4.5; возвращает план назначений с обоснованием каждого решения. Модуль должен поддерживать три режима работы: (а) одиночный запрос - для конкретной заявки выдать ранжированный список кандидатов с объяснением выбора; (б) пакетный запрос - для набора заявок на смену/день найти глобально оптимальное распределение техники, включая решение о том, какие заявки целесообразно объединить в multi-stop маршрут; (в) оценка группировки - для произвольного набора заявок рассчитать, какие из них выгодно объединить в один выезд, а какие обслуживать раздельно, с возвратом экономии относительно baseline. Подход к реализации выбирается участниками свободно (см. раздел 3.4). Приветствуется реализация наивного baseline-алгоритма (например, жадное назначение ближайшей свободной техники) для количественного сравнения с основным решением.


7.5. Функция скоринга и объяснимость
Каждая рекомендация, возвращаемая API, должна сопровождаться числовым score и текстовым обоснованием (reason). Участникам необходимо разработать формулу скоринга, которая агрегирует факторы, перечисленные в целевой функции (раздел 3.2), в единую метрику сравнения кандидатов. Формула должна быть документирована: какие веса присвоены расстоянию, времени ожидания, приоритету, штрафам за нарушение SLA и совместимости. Поле reason является опциональным и реализуется как бонусная функциональность: для его формирования рекомендуется использовать LLM, которая на основе числовых параметров выбора входа и выхода генерирует объяснение на естественном языке.
7.6. Визуализация маршрутов
API обязан возвращать геометрию маршрута в виде упорядоченного списка координат, пригодного для отрисовки polyline. Для демонстрации необходимо реализовать визуальное отображение маршрутов на карте (Leaflet, Folium, Kepler.gl или аналог). Визуализация должна показывать: стартовую позицию техники, точку назначения (скважину), линию маршрута по дорожному графу и, в случае multi-stop маршрута, промежуточные точки с порядком обхода.

Сервис принимает список заявок, оценивает целесообразность объединения и возвращает оптимальную группировку: какие заявки выгодно объединить в multi-stop маршрут, а какие обслуживать отдельно. Поля baseline_distance_km и baseline_time_minutes содержат суммарные показатели при раздельном обслуживании каждой заявки, savings_percent - процент экономии предложенной группировки относительно baseline. Параметр max_detour_ratio ограничивает допустимое удлинение маршрута при объединении (например, 1.3 - не более 30% крюка), max_total_time_minutes - максимальное суммарное время выезда. Если объединение ни одной пары заявок не даёт выигрыша в рамках constraints, возвращается strategy_summary: "separate" с каждой заявкой в отдельной группе.

10.1. Работающий прототип
Backend + API, тестируемый через Postman/curl или простой фронтенд. Код выложен на GitHub/GitLab с README (как запустить, примеры запросов).
10.2. Демонстрация на 2-3 сценариях
• Сценарий 1 (срочная заявка): high priority → система выбирает оптимальную технику с обоснованием, показывает маршрут.
• Сценарий 2 (сравнение): medium priority → "просто ближайшая техника как baseline" vs улучшенный алгоритм → визуализация разницы в километраже/времени.
• Сценарий 3 (многозадачность): 3 заявки в одном районе → одна машина vs три отдельных, с обоснованием и цифрами.
